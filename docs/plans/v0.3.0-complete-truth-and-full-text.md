# v0.3.0 — Immutable Document Foundation + Mutable Context Layer

> **Framing (the two layers, built in order).**
>
> **Layer 1 — the immutable foundation. Build it structure-first, top-down.** Before content or
> interpretation matters, establish the skeleton: *what is the **case**, what is each **filing**,
> what **documents** are in each filing* — then explode every document into **atomic, fully
> searchable parts**. The documents and their explosion are **absolute and immutable**: ground truth
> that never changes. This is the foundation everything else rests on.
>
> **Layer 2 — the mutable context layer, on top.** *Where* each document came from (`ody` =
> Odyssey, `txe` = re:SearchTX), *how and when* it was filed, the timeline, and the cross-source
> reconciliation into one optimized "complete truth." This layer is **derived** from the foundation
> and is **mutable** — it can be re-reconciled, corrected, and enriched as sources improve, **without
> ever mutating the frozen foundation beneath it.**
>
> **Scope guard.** *Schemafy what exists.* No analysis dimensions (claims / deadlines /
> relationships inferred from PDF text) — those are M3+. The only machine-generated text is OCR for
> image-only pages, already flagged non-evidential. "Atomic part" = **page** for this milestone (the
> citable legal unit); sub-page (paragraph) splitting is deferred until the skeleton is solid.
>
> **Location.** Case data lives in the separate `gcp-hoa-case` repo; lawnlord runs against its
> intake folder (`../gcp-hoa-case/intake/combo`). Case data never lives in the tool repo.

## Context — where the code is today

v0.2.0 ships the substrate: `Case.from_intake()` → explode → ingest → DuckDB index
(`cases / parties / events / documents / document_events / sections / chunks`) → `query` with
provenance, optional OCR, and a canonical `case.json` + `lawnlord pack`. Against the real `combo`
intake: **22 documents → 37 sections → 255 page-chunks, 0 page-count mismatches, every page has
text.** 125 tests pass.

Two gaps between today's code and the model above:

- **The filing is collapsed into the document.** The exploder treats one PDF as its own
  "submission" (`submission_slug == document_slug`). But the real docket has **filings that contain
  multiple documents** — verified in this case: Notice of Submission → **3** documents, Docket
  Control Order → **2**, Notice of Pending Accounting → **2** (the other 17 filings hold 1 each; no
  document spans two filings). The data exists today only as a generic `document_events` join;
  this milestone promotes **Filing** to a first-class structural container in the model and schema.
- **"Fully searchable" isn't guaranteed.** `query --text` is a substring scan (`LIKE '%…%'`), not a
  ranked full-text index, and OCR is opt-in — so a scanned filing would explode into empty,
  unsearchable pages.

### The hierarchy this milestone makes explicit

Confirmed against the source data (not assumed) — Odyssey's own field name for the filed PDF is
`image`, and a single image bundles multiple logical documents:

```
Case  (the matter: 25-09-14566)
└── Filing  (a docket submission/event — what was filed, when, by whom)   ← promoted to first-class
    └── Image  (the filed PDF — Odyssey's term; immutable, hash-pinned artifact)   ← NEW level
        └── Document  (a logical document inside the image: Motion, Exhibit A, Affidavit…)   ← NEW level
            └── Section / divider  (sub-structure within a document; nested bookmarks/headings)
                └── Page  (the atomic, citable, fully-searchable part; immutable)
```

**Vocabulary correction.** Today's code is off by one: it calls the **PDF file** a "document"
(should be **image**) and the **logical units inside it** "sections" (should be **documents**). This
milestone realigns the model to the source's vocabulary.

**Source evidence (case `25-09-14566`):**
- *Filing → 1+ images:* Notice of Submission = 3 images, Docket Control Order = 2, Notice of Pending
  Accounting = 2; 17 filings = 1; some pure docket entries = 0.
- *Image → documents within:* `Plaintiffs_Motion_for_Summary_Judgment.pdf` (one image, 121 pages)
  bookmarks into Motion + Exhibits A/B/C/C-1/C-2 + Attorney's-Fees Affidavit + Exhibits D-1/D-2 —
  **8–9 distinct documents in one file.** 8 of 22 images split into >1 document today.
- *Messiness to absorb:* only some images carry bookmarks (the rest fall to heading-scan or are
  single-document); bookmark trees include junk (embedded-file refs `MSJ Acct Statement.pdf`,
  `209.pdf`, page `-1`, nested numeric IDs) that are **not** real documents.

Layer 2 (context) wraps this with reconciled identity, source provenance, the timeline, conflicts,
and gaps — mutable, derived, never altering the tree above.

## Phase 1 — The immutable foundation (structure-first, top-down)

### The proof: lossless explode ↔ reassemble round-trip

**The foundation is correct if and only if we can reassemble the entire exploded case back into one
PDF without losing any context.** This is the headline acceptance test for all of Phase 1 — if
reassembly drops a page, reorders, flattens the Filing → Image → Document hierarchy, or loses an
embedded attachment, the explosion was lossy. "No context lost" means **content** (every page,
exactly once, in docket order) **and structure** (the hierarchy + provenance survive in the rebuilt
PDF's outline + sidecar manifest).

A proof-of-concept against real `combo` already passes the core of this: 22 images → one 255-page
master PDF; page count == Σ source pages; **every page's text identical to its source, page-for-page**;
a 61-entry Filing → Image → Document outline; every master page traceable back to (image,
source_page). What the PoC did *not* yet guarantee — and the shipped test must:

- **Byte/visual page fidelity**, not just text. Reassemble from the **preserved original images**
  (hash-pinned artifacts) so page bytes come from the immutable source, not a re-encode.
- **Embedded attachments / annotations / signatures.** The MSJ exhibits carry embedded files
  (`MSJ Acct Statement.pdf`, `209.pdf`) that a naive page-copy drops — these are context and must
  survive (or be explicitly inventoried) in the round-trip.

Deliver this as `lawnlord assemble [--case-dir …] -o case-master.pdf` (builds the master + outline +
a page→provenance manifest) plus a round-trip integrity test asserting: Σ source page hashes ==
master page hashes, full hierarchy reproduced, and every embedded artifact accounted for.

- **Acceptance:** `assemble` over real `combo` yields a 255-page master whose pages are visually +
  text identical to the originals in docket order; the outline reproduces Filing → Image → Document;
  every page and every embedded attachment traces back to a source image by hash; a dropped page,
  reorder, or lost attachment fails the test.
- **Label:** `enhancement`

### F1 — Case (the skeleton root)

Establish the case identity as the deterministic root of the tree from the intake metadata
(caseNumber is the key). Just enough to anchor filings/images — *not* the reconciled
complete-truth identity (that's Layer 2).

- **Acceptance:** `combo` intake yields exactly one case keyed by `25-09-14566`; the foundation
  build never invents a field the metadata doesn't state.
- **Label:** `enhancement`

### F2 — Filing as a first-class container (of images)

Promote the docket submission to a real entity: a **Filing** has a date, a type/title, the filing
party, and **contains one or more Images**. Source it from the docket (Odyssey `timeline.files[]` /
re:SearchTX `events.documents[]`). Replace the "1 PDF = 1 submission" assumption.

- **Acceptance:** against real `combo`, the Notice of Submission filing contains its 3 images, the
  DCO filing its 2, Notice of Pending Accounting its 2; the remaining filings hold 1; the
  `case → filing → image` tree round-trips; a filing with no image (pure docket entry) is
  representable.
- **Label:** `enhancement`

### F3 — Image: the immutable, referenced artifact (the filed PDF)

The **image** is the filed PDF — Odyssey's own term — and the unit of preservation. Referenced by
`intake_path`, pinned by `sha256`, with `declaredPageCount` + `actualPageCount` +
`pageCountMismatch`. Originals are never opened for writing. Immutability is an explicit schema
guarantee, not just a DB detail (`lawnlord pack` already pairs `case.json` with `filings/`).
(This is what today's code mislabels "document".)

- **Acceptance:** every image record has a non-empty `sha256` and both page counts; a tampered byte
  changes the hash and is flagged; no code path opens a source PDF writable; images dedupe by
  content hash; the 121-page MSJ image records its real page count.
- **Label:** `enhancement`

### F4 — Document: the logical document(s) inside each image  ✅ DONE

> **Shipped.** The DuckDB index re-leveled to `images` (filed PDFs) + `documents` (the logical
> documents within), with `chunks` carrying both ids; query API renamed; `SCHEMA_VERSION` → 2. The
> exploder's on-disk "section" vocabulary is mapped to `documents` at the index boundary. Verified:
> the MSJ image → its 9 bookmarked documents, junk excluded; 130 tests green. See the CHANGELOG.

Explode each image into the **documents within it** — Motion, Exhibit A, Affidavit, etc. — using
the existing four-tier boundary detection (bookmarks → heading-scan → fallback), re-leveled from
"section" to **document**. Must be robust to source messiness: images with no bookmarks fall to
heading-scan or resolve to a single document covering 1..N; junk bookmarks (embedded-file refs,
`page -1`, nested numeric IDs) are not promoted to documents. Sub-structure within a document
(nested headings/dividers) is retained as **sections** beneath it. Gapless page coverage preserved;
low-confidence splits stay flagged `needsHumanReview`.

- **Acceptance:** the MSJ image explodes into its bookmarked documents (Motion + Exhibits A–D +
  Affidavit), each with a contiguous page range; an image with no bookmarks yields one document
  spanning all its pages; junk/embedded bookmarks are excluded (or flagged), not emitted as
  documents; `case → filing → image → document` round-trips; re-explode is byte-identical.
- **Label:** `enhancement`

### F5 — Page: atomic explosion, every page fully searchable

Explode each document into its atomic parts (pages), guaranteeing **every page lands real,
searchable text** — the heart of "perfect explosion before content matters":

- **Auto-OCR image-only pages.** Pages with no extractable text (`ocrLikelyNeeded`) are OCR'd by
  default during the build so no page is left blank; OCR text keeps `textSource=ocr` +
  `ocrConfidence` and stays machine-generated / non-evidential. A `--no-ocr` escape hatch preserves
  the fast path. Born-digital pages are untouched (this packet: 0 image pages — provably unchanged).
- **Real ranked full-text search.** Replace the `LIKE` scan with a DuckDB full-text-search index
  over page text (`PRAGMA create_fts_index` → BM25 `match_bm25`), built deterministically during
  `index` and rebuilt on re-index. `query --text` keeps its provenance columns (document title,
  source page, citation) and gains relevance ranking + multi-term/phrase matching. **Best-effort
  with the existing substring search as a logged fallback** if the FTS extension can't load (no hard
  dependency, works offline) — per your "what's FTS" check, FTS = the indexed, ranked search engine
  over the page text.
- **Integrity guard preserved.** Gapless 1..N page coverage; declared-vs-actual page-count
  cross-check is surfaced, not hidden.

- **Acceptance:** `count(pages)` == total exploded pages with no gaps; a synthetic image-only page
  yields a non-empty `textSource=ocr` chunk by default (and an empty `needs_review` chunk under
  `--no-ocr`); `--text "summary judgment"` returns BM25-ranked rows with provenance; a multi-term
  query matches; re-index is byte-identical.
- **Label:** `enhancement`

## Phase 2 — Unify the two mirror views into the standard court schema

> **Reframed.** `txe` and `ody` are **two views of the same court record**, not competing sources
> (verified: identical caseNumber/title/judge/parties; differences are date/name formats, court-vs-
> clerk facets, and view taxonomies). So this phase is **mirror-unification + normalization** into
> the court-defined standard — *not* conflict reconciliation. The court defines the schema down to
> the **image**; this phase makes `case.json` mirror it faithfully and completely. The full field
> map is **[`docs/standard-schema.md`](../standard-schema.md)**.
>
> **Sequencing caveat (per the image-is-a-container model).** This standard schema (down to image)
> and the image→pages explosion (F5 + the assemble round-trip) are the *must-be-perfect-first*
> foundation. Breaking images into documents-within (F4, already shipped) is **additive** and sits
> below this — it is not a prerequisite for, and must not block, perfecting the down-to-image schema.

Derive `case.json` to the standard above (**`SCHEMA_VERSION` 1.0 → 2.0**): field-complete (nothing
dropped), ISO/canonical-normalized, each value tagged with the view(s) that supplied it. It
references the foundation by case/filing/image/page IDs and never rewrites it.

### What the sources expose (field inventory, case `25-09-14566`)

| Source file | Fields |
| --- | --- |
| `ody/case-summary.json` | caseNumber, **citationNumber**, caseTitle, dateFiled, location, caseType, status, sourceUrl |
| `ody/case-history.json` | identity (court, judicialOfficer), parties[]+attorneys[name,status,phone], **disposition{date,type,judicialOfficer,comment,files}**, timeline[20]{date,phase,event,description,files}, financialInformation{assessment,payments,balanceDueAsOf}, **sources[]** |
| `ody/register-of-actions.json` | parties[]+**representation**+**location**, dispositions[], **otherEventsAndHearings[26]**{date,description,files}, financialInformation.**transactions[]**{date,description,amount} |
| `ody/filings.json` | selectedEvent / otherEventsOnThisCase / otherImagesOnThisCase → {date, event, image(=title), pageCount, file} |
| `txe/meta.json` | _meta{source, caseUrl, **extractedNote**}, caseInformation{caseTitle, caseNumber, **caseLastRefreshed**, location, **caseCategory**, caseType, caseFiledDate, judge, caseStatus}, **caseFlags**, **caseCrossReferences**, parties.rows[type,name,**nicknameAlias**,attorneys[name,**attorneyNumber**]], hearings.rows[2]{dateTime,hearingType,judge,location,result}, events.rows[26]{row,date,event,type,comments,documents[name,pages]} |

**Bold = available but dropped by today's canonical schema.**

### Unification cases (the test fixtures) — view differences, not conflicts

- **View taxonomy — `caseType`.** ody = `Foreclosure - Other Foreclosure`; txe = `Other Civil` /
  `Civil - Other Civil`. Rule: keep the more specific as `caseType`, the other as `caseTypeAlt`.
- **Wording — `status`.** ody = `Disposed`; txe = `Closed (Disposed)`. Normalize to a canonical
  token; keep raw per view.
- **Venue facets — `court` vs `clerk`.** `284th District Court` and `Montgomery County - District
  Clerk` are distinct fields, both kept.
- **Pagination, not divergence — 20 vs 26 events.** case-history timeline = 20 phase-tagged; register
  + txe = 26 (txe: "paginated 20/page, total 26"). Standard filing list = the **union**, phase-tagged
  where Odyssey provides it.
- **Format normalization.** dates `09/05/2025`/`9/5/2025` → ISO `2025-09-05`; names
  `John William O'Grady`/`O'Grady, John William` → canonical `Last, First` + aliases. Raw retained.

### C-tasks

- **C1 — Field-complete source readers (nothing dropped).** Every bold field gets a home; a
  "no silent drop" test asserts each inventoried source key is represented or explicitly ignored.
- **C2 — Unifier (`unify.py`).** N mirror-view source-models → one standard `CaseModel` matching
  `docs/standard-schema.md`: documented normalization rules (ISO dates, `Last, First` names,
  most-specific caseType + `caseTypeAlt`, canonical status, court/clerk both kept) and **per-field
  source tags** `{value, sources:[ody|txe], raw}`. Because the views mirror one record, a genuine
  value disagreement is the rare exception (data corruption) — surfaced in `gaps`, not treated as a
  routine `conflicts[]` flow. `combo` = "read every view present → unify"; degrades to one reader
  when only one view exists.
- **C3 — Canonical schema v2.0.** Rewrite `to_canonical`/`from_canonical` to the standard: superset
  (nothing dropped) + source tags + `gaps[]` + source-freshness; lossless round-trip on the fully
  populated real model.
- **C4 — Land the standard in DuckDB.** Idempotent schema bump for the new identity fields
  (citationNumber, disposition comment, caseTypeAlt, clerk, freshness), `financial_transactions`,
  and `gaps`; existing `query` flags unaffected.

## Phase 4 — The capstone: the case bundle (the final output)

**The final output is one self-contained case bundle: the complete exploded case content, wrapped
inside the standard contextual metadata, fully cross-linked.** The metadata (the court schema down
to image, `docs/standard-schema.md`) is the outer wrapper and index; the content lives inside it;
every record links both ways. This is the unit lawnlord ships and that AI analysis consumes —
context-complete, portable, and grounded (every page traceable to the court record).

It composes the three pieces that already exist separately:

- `lawnlord pack` → `case.json` (the metadata wrapper) + preserved `filings/` (immutable artifacts).
- `lawnlord assemble` → the complete case bundled into one master PDF + page→provenance manifest.
- `lawnlord index` → the searchable content (DuckDB + per-page text).

### Bundle shape (one zip / directory)

```
<case>.bundle/
  case.json              # STANDARD metadata wrapper (court schema down to image) — the index
  images/                # preserved original PDFs (immutable, hash-pinned)        — court leaf
  case-master.pdf        # the complete case, reassembled in docket order + outline (lossless)
  pages/                 # per-page extracted text (fully searchable)              — image→pages
  lawnlord.duckdb        # the queryable index (optional, regenerable)
  bundle-manifest.json   # cross-links: every image ↔ its pages ↔ master-PDF pages ↔ filing
```

Each `case.json` image record carries its `sha256`, its page range in `case-master.pdf`, and the
path to its pages/text; each page traces back up to its image, filing, and timeline position. The
bundle round-trips: it contains everything needed to reassemble the complete case **and** to read
its full context, with no external dependency.

- **Outcome:** `lawnlord bundle <intake> -o <case>.bundle.zip` produces the wrapped, complete,
  cross-linked case.
- **Acceptance:** opening the bundle and reading only `case.json` reaches every image, page, and the
  master PDF; the lossless reassembly proof holds from inside the bundle; the bundle is
  self-contained (no path escapes it) and regenerable byte-for-byte.
- **Label:** `enhancement`
