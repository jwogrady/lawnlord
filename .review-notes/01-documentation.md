# Documentation Reviewer

**Date:** 2026-06-25
**Reviewer:** jwogrady

## Finding

lawnlord has an unusually *rich* documentation surface for an alpha solo project — a polished README, a disciplined doc model (`contributing.md` codifies a "code wins" rule and a README/ROADMAP/CHANGELOG/LICENSE root contract), a real Keep-a-Changelog history, a prerequisite-chain ROADMAP wired to GitHub issues/milestones, ten ADRs, and two deep problem statements. The *intent* is excellent and the writing is clear. The fatal flaw is **truth drift**: the three documents explicitly designated as the authoritative "developer code summaries" (`architecture.md`, `schema.md`, `ux.md`) have fallen badly out of sync with the code they promise to mirror, and the README's "what exists today" section describes a teardown state the repo left weeks ago. The project preaches "if a doc and the code disagree, the code wins — fix the doc" (`contributing.md:36`, `architecture.md:4`) and then violates that rule in its three flagship docs.

The drift is not subtle. `architecture.md:42` and `schema.md:58` both state `SCHEMA_VERSION = 6`; the code is at **11** (`db.py:48`). `architecture.md:28` says "One subcommand registered in `cli.py`: `start`" and `ux.md:18` tables only `start` as callable "today"; the parser actually registers **ten** subcommands — `start, import, export-actual, export-exploded, export-metrics, export-regions, regions, explode, transcribe, measure` (`cli.py:87–223`). `architecture.md:39` and the README (`README.md:54`) call `workspace.Case.from_intake` a `NotImplementedError` stub awaiting "the next branch" — it is fully implemented and calls `reader.load_case_model` (`workspace.py:49–75`). The schema docs list four "not yet populated" tables (`chunks`, `extracted_dates`, `knowledge_documents`, etc.) that **no longer exist** in `db.py` (the real new tables are `pages`, `page_text`, `page_regions`). The entire web viewer (~2,094 LOC, a documented product surface in the CHANGELOG) is invisible in `architecture.md` and `ux.md`. Net effect: the docs describe roughly the **0.4.0 teardown snapshot**, while the code has shipped the F1–F5 foundation plus transcription, regions, and confidence gauges on top of it.

The most quotable instance of drift is the README test badge: it advertises **"tests — 27 passing"** (`README.md:8`, `README.md:105`), but `uv run pytest --collect-only` reports **94 tests collected** across 11 files. Two other dead references compound the credibility hit: ROADMAP principle #2 cites enforcement "by the `ALLOWED_CURATED_FIELDS` whitelist (`curation.py`)" (`ROADMAP.md:42`) — there is no `curation.py` and no `ALLOWED_CURATED_FIELDS` anywhere in `src/` — and `motivation.md:53` still tells a reader "Today `lawnlord start` scaffolds the intake" as if it were the only working command. By contrast, the **CHANGELOG is exemplary and current** (the `[Unreleased]` block accurately narrates #125–#129 down to the `flaggedPageDetails` field), which makes the staleness of the code-summary docs all the more glaring: the author keeps the *history* doc honest but lets the *reference* docs rot.

## Evidence

- **README badge vs reality — `README.md:8` and `README.md:105`:** badge `tests-27 passing` and "characterization suite (27 tests)"; actual `pytest --collect-only` = **94 tests collected** (11 files). A 3.5× understatement on the project's headline trust signal.
- **`README.md:42–54` ("What's in the repo today"):** describes only "the zip-view studs" and says `workspace.from_intake` "is a stub awaiting the zip → CaseModel reader (the next branch)." Code: `workspace.py:49` `from_intake` is implemented and loads/validates via `reader.load_case_model`. The README omits `reader.py`, `export.py`, `transcribe.py`, `regions.py`, `explode.py`, and the whole `web/` app — all shipped.
- **`docs/architecture.md:42` & `docs/schema.md:58`:** assert `SCHEMA_VERSION = 6`. Code: `db.py:48` `SCHEMA_VERSION = 11`. The schema docs' "present but not yet populated" tables (`chunks`, `extracted_dates`, `knowledge_documents`) do not exist in `db.py`'s `CREATE TABLE` set (`db.py:52–193`); the real additive tables `pages`, `page_text`, `page_regions` are undocumented in both summary docs.
- **`docs/architecture.md:28` & `docs/ux.md:14–18`:** claim `start` is the only registered/callable subcommand. Code: `cli.py:85–223` registers `start, import, export-actual, export-exploded, export-metrics, export-regions, regions, explode, transcribe, measure` (10).
- **`docs/architecture.md` & `docs/ux.md` — web app absent:** neither mentions the Bun viewer, its lenses, or routes (`/api/case`, `/api/exploded`, `/api/metrics`, `/api/regions`, `/png/*`, port 4173). The only place the web app is documented is `web/CLAUDE.md` (good, but isolated, and references CASE_DIR/`bun dev` flow not surfaced anywhere a non-web reader would look).
- **`ROADMAP.md:42` dead reference:** "the `ALLOWED_CURATED_FIELDS` whitelist (`curation.py`)." No such file/symbol exists (`grep` of `src/lawnlord/` returns nothing). Documents an enforcement mechanism that is not in the code.
- **`docs/motivation.md:53`:** "Today `lawnlord start` scaffolds the intake. The zip → DuckDB reader and the two views … are the prerequisite chain in the ROADMAP" — stale; import + transcribe + both views shipped in 0.4.0+.
- **ADR statuses out of step with code — `docs/adr/0001:3`, `0002:3`, `0003:3`:** all `Status: Proposed`, yet the local-first transcription decision they describe is **implemented**: `transcribe.py` has `LocalTranscriber` (Ollama) and `LlamaCppTranscriber` (GPU), and the active branch `feat/llamacpp-gpu-vision` just committed it (c1a4490). Proposed ADRs describing shipped code should be Accepted. (0004–0009 are correctly Accepted; 0004 cleanly notes its supersession by 0006.)
- **No formal CLI/API spec — confirmed gap:** `grep` finds no OpenAPI/JSON-schema/reference doc for the `export-*` / `transcribe` / `import` commands; the only contract is `argparse` `help=` strings in `cli.py` and the read-models in `export.py`. ADR-0007 (`0007:8`) references `export-actual`/`export-exploded` in prose but no doc enumerates flags, JSON output shapes, or the web routes. A new dev must read `cli.py` + `export.py` + `web/index.ts` to learn the surface.
- **Positives:** `CHANGELOG.md:13–67` `[Unreleased]` is accurate and detailed (matches #125–129 in code, down to `flaggedPageDetails`); `contributing.md` is a strong, opinionated governance doc (doc model, invariants, no-AI-attribution rule, branching); `problem-transcription-*.md` are sharp, well-scoped problem statements; `motivation.md` (apart from the stale line) is a compelling why; ADRs 0004–0009 are genuinely useful design records.
- **No root `CLAUDE.md`/`AGENTS.md`:** only `web/CLAUDE.md` exists. The README links `docs/` well, but there is no top-level agent/onboarding contract for the Python side (the engineering invariants live in `contributing.md`, which is good, but not where an agent tool looks first).

## Scoring

**Dimension:** Documentation / onboarding
**Score:** 5
**Rationale:** The documentation *architecture* is genuinely strong (8-quality README prose, disciplined doc model, real ADRs, an exemplary CHANGELOG), but the three docs the project itself designates as authoritative code summaries are materially false — wrong schema version, wrong command set, "stub" claims for shipped code, a phantom `curation.py`, and a headline test badge off by 3.5×. A doc that confidently states the wrong thing is worse than no doc, because a new developer would trust it; the project's own "code wins, fix the doc" rule is broken in exactly the files meant to enforce it. Strong bones, decayed flesh — a 5, not lower only because the README/CHANGELOG/ROADMAP/ADR-0004–0009 spine still onboards the *intent* well.
