## Finding

lawnlord's architecture is genuinely good, and the goodness is not accidental — it is the most coherent dimension of the project. The data flow is a clean unidirectional pipeline (zip → `reader.load_case_model` → `models.CaseModel` → `ingest.ingest_case` → DuckDB → `export.*` → web), and the canonical/derived/analysis separation is real rather than aspirational. Modules have single, legible responsibilities (`db.py` is the *only* SQL site that defines schema; `reader.py` is the only zip parser; `intake.py` is the only place that knows the folder layout; `ingest`/`explode`/`transcribe`/`regions` each own exactly one additive layer), the public surface is re-exported flat from `__init__.py`, and the layering is documented *in the schema itself* — `db.py:31-47` narrates v7→v11 as "mirror, then Exploded, then page_text, then page_regions, all additive." The CLI/web split is the cleanest expression of the invariant: the web tier (`web/index.ts`) holds zero SQL and zero zip parsing, reaching every dynamic datum through `uv run lawnlord export-*`, so a read-only viewer *structurally cannot* mutate the record. Determinism is enforced by construction — content-hash IDs (`doc_<sha16>`, `pt_<sha16>`, `pr_<sha16>`), a deterministic `generated_at` from the manifest's `capturedAt` (never wall-clock; `reader.py:43,100-109`), and fixed-scale PNG renders (`explode.py:24,57`) — so re-import is byte-identical and idempotent.

The immutability invariant, however, is enforced **partly structurally and partly by convention**. Structural enforcement is strong where it counts: the mirror is rebuilt by `_clear_case` + re-insert (`ingest.py:44-58`) so the zip is always the authority; `page_text`'s surrogate-id PK (ADR-0005) makes rev-0 of any `(page_id, source, model)` variation collide on re-insert, which is a real database-level append-only guard; and `regions.capture_pdf_regions` refuses to emit a box unless the stored anchor text *exactly* matches a fresh PDF read (`regions.py:173`), so geometry is never fabricated or misaligned. But the additive layers' "never mutate the mirror" promise rests on the writers being well-behaved, not on a permission boundary — every write path opens the same DB read-write with `apply_schema` (`cli.py:436-437,460-461,491-492`), and nothing prevents a future `analysis` module from issuing `UPDATE images`. The mirror is immutable because the code chooses not to touch it, enforced at code-review time, not by the engine. The one true structural backstop is the read-only CLI export path the web tier is confined to.

The real architectural debt is twofold. First, **schema versioning is decorative**: `apply_schema` (`db.py:231-237`) runs `CREATE TABLE IF NOT EXISTS` for every table and writes `SCHEMA_VERSION` into `schema_meta` *only when the table is empty* — it never reads the stored version, never compares it to `SCHEMA_VERSION`, and has no migration path (`db.py:38` openly states "bumps need no in-place migration" because DBs are regenerable). The failure mode is silent: open a v9 DB with v11 code and you get the *new* tables (page_regions etc.) added but the *old* recorded version left at 9, and any changed column on a pre-existing table silently ignored — a stale-schema DB that reports a wrong version and never warns. "Regenerable" is only safe if the operator knows to delete and rebuild, and nothing tells them. Second, the **web shell-out boundary, while clean for correctness, is a scalability and error-handling liability**: every `/api/*` request cold-starts a `uv run` Python process + DuckDB open + a *full-case* export (`web/index.ts:63-124`, `download.ts:110`), with no caching, no pooling, and inconsistent error handling — `/api/metrics` and `/api/regions` catch the subprocess failure and return `{error}`+500, but `/api/case` and `/api/exploded` do not, so a CLI nonzero exit there becomes an unhandled rejection / opaque 500. The `from_intake` docstring/module header in `workspace.py:6-8` still says "the reader … lands on the next branch" and "creates nothing" — stale narration over fully-implemented code (the doc-drift Agent 01 flagged, now confirmed at the source).

## Evidence

- **Clean unidirectional data flow:** `reader.load_case_model` (`reader.py:227-240`) → `models.CaseModel` (`models.py:217-231`) → `ingest_case` (`ingest.py:61-197`) → DuckDB (`db.py`) → `export.export_*` (`export.py:172-609`) → web (`web/index.ts:63-124`). No back-edges; the web tier never imports Python modules, only shells the CLI.
- **Single-responsibility module boundaries hold:** SQL schema lives *only* in `db.py:51-211` ("the only SQL site," `db.py:3`); zip parsing only in `reader.py`; folder-layout knowledge only in `intake.py:16-19` ("the single place that knows the layout"); each additive layer in its own module (`explode.py`, `transcribe.py`, `regions.py`).
- **Layer separation is documented in the schema and honored in code:** `db.py:31-47` narrates mirror (7 tables) → Exploded (`documents`,`pages`) → `page_text` → `page_regions` as strictly additive; `explode.py:49-50`, `transcribe.py` `_PAGE_TEXT_INSERT` (`transcribe.py:72-75`), and `regions.py:131` only ever DELETE/INSERT *their own* layer's rows, never the mirror's `cases`/`parties`/`events`/`images`.
- **Append-only is a real DB guard, not just convention:** `page_text` surrogate PK = hash of `(page_id, source, model, rev)` (`transcribe.py:78-86`, ADR-0005) → re-inserting an identical variation conflicts on PK, so rev 0 is immutable; `_next_rev` (`transcribe.py:488-498`) appends within a variation. Same pattern for `page_regions` ids (`regions.py:78-84`, hashing *structure only*, never coordinates).
- **Determinism by construction:** content-hash IDs `doc_<sha16>` (`ingest.py:157`), `pt_<sha16>`, `pr_<sha16>`; `generated_at` from manifest `capturedAt` with a fixed sentinel fallback `1980-01-01T00:00:00Z` (`reader.py:43,100-109`) — never wall-clock; fixed-DPI deterministic PNG render (`explode.py:24,57,83`).
- **CLI/web read-only boundary enforces immutability structurally:** `web/index.ts` has no SQL and no zip parse; exports run `uv run lawnlord export-*` (`index.ts:64,74,87,109`; `download.ts:110`). The CLI opens the DB `read_only=True` for every `export-*` (`cli.py:389,399,415,425,561`), so the viewer path *cannot* write.
- **Schema versioning is decorative (debt):** `apply_schema` writes `SCHEMA_VERSION` only `if row[0] == 0` and never reads/compares it (`db.py:231-237`); no `ALTER`/migration anywhere in `src/` (grep confirms only `db.py:38` "no in-place migration"). A stale DB opened by newer code gets new tables via `CREATE TABLE IF NOT EXISTS` but keeps the old recorded version and silently ignores changed columns — no detection, no warning.
- **Mirror immutability is convention past the export boundary:** all write commands open the DB read-write and `apply_schema` (`cli.py:436-437,460-461,491-492`); nothing stops a future writer from `UPDATE`-ing a mirror table. The "additive only" rule is enforced by code review, not by the engine.
- **Per-request subprocess spawn — scalability ceiling:** every `/api/*` hit cold-starts `uv run` + Python import + DuckDB open + a *whole-case* export with no cache/pool (`index.ts:63-124`, `download.ts:106-118`). Fine for a single-user local desktop tool (its stated scope, `index.ts:1`), but it breaks at multi-user, large-case, or rapid-navigation scale; the full-case `export_exploded` is recomputed on every page view.
- **Inconsistent web error strategy:** `/api/metrics` (`index.ts:85-103`) and `/api/regions` (`index.ts:106-124`) wrap the subprocess in try/catch → `{error}`+500 graceful degradation; `/api/case` (`index.ts:63-71`) and `/api/exploded` (`index.ts:73-81`) do **not** — a CLI failure there is an unhandled `Bun.$` rejection / opaque 500 with no `{error}` contract. Python side is consistent: `main` catches `FileNotFoundError`/`ValueError` → clean exit 1, lets unexpected errors surface (`cli.py:321-329`).
- **Single-writer / lock contention risk:** DuckDB is single-writer-per-file; `explode`/`transcribe`/`regions` hold a read-write handle for the whole run while the viewer may concurrently `uv run export-*` (read-only is fine, but a write command during a read, or two writes, will lock). No locking strategy is documented or coded.
- **Testable seams are good where they matter:** `Transcriber` Protocol (`transcribe.py:229-237`) + injectable `client` on `CloudTranscriber` (`transcribe.py:244-245`) make the network tier fully mockable; pure functions `agreement_score`/`divergence_spans`/`_consensus_index` (`export.py:67-126`) are trivially unit-testable. Harder-to-test seams: anything touching `pypdfium2` rendering (`explode.py`, `regions.extract_pdf_charboxes`) needs real PDFs; the `urllib`-based `LocalTranscriber`/`LlamaCppTranscriber` HTTP calls have no injected transport (only the Protocol seam above them).
- **DRY is well-observed:** one `_PAGE_TEXT_INSERT` shared by transcribe + escalate (`transcribe.py:72-75`); `_ollama_tags` shared by `ollama_available` + `installed_vision_models` (`transcribe.py:354-399`); divergence/agreement defined once in `export.py` and *reused* by both the exploded lens and `export_metrics` (`export.py:482` "so metrics and the viewer can never disagree") — a deliberate single-source-of-truth choice.
- **Stale module narration (confirmed doc-drift at source):** `workspace.py:6-8` header — "The reader … lands on the next branch; this module … creates nothing" — over `from_intake` which is fully implemented and calls `load_case_model` (`workspace.py:48-77`). `OUTPUT_SUBDIRS` (`workspace.py:27-35`) lists 7 dirs (`artifacts`,`knowledgebase`,`analysis`,`outputs`,`manifests`) that no current code writes to — inherited scaffold from the pre-pivot design.
- **`explode_case` single-case assumption:** picks `case_id` via `SELECT id FROM cases LIMIT 1` (`explode.py:44-47`) — correct for the one-case-per-DB model, but an implicit coupling to that invariant that a multi-case DB would silently violate (it would explode only the arbitrary first case).

## Scoring

**Dimension:** Architecture / sustainability
**Score:** 8
**Rationale:** This is a textbook-clean layered architecture for its stated scope — crisp module boundaries, a real (if convention-backed) additive-layers discipline, determinism by construction, and a CLI/web split that makes the read-only viewer structurally incapable of mutating the record. It loses points for decorative schema versioning with a silent stale-DB failure mode, a per-request subprocess model and inconsistent web error handling that cap scalability and robustness, and stale module-level narration that betrays the mid-rebuild state — all sustainability concerns rather than design flaws, so an 8, not lower.

## Notes to Next Agent

- **For Code Quality (03):** hard-to-test seams are the `pypdfium2` paths (`explode.py`, `regions.extract_pdf_charboxes`) and the raw-`urllib` HTTP in `LocalTranscriber`/`LlamaCppTranscriber` (`transcribe.py:267-352`) — only the `Transcriber` Protocol above them is mockable, the transport itself is not injectable. The `_parse_local_output` salvage logic (`transcribe.py:200-226`, regex over malformed JSON) is fragile and deserves its own scrutiny. Stale narration to flag for cleanup: `workspace.py:6-8` and the unused `OUTPUT_SUBDIRS` (`workspace.py:27-35`).
- **For Testing (04):** the state-management complexity worth targeting is **schema drift**: `apply_schema` (`db.py:231-237`) never validates the stored `schema_meta.version` against `SCHEMA_VERSION`, so there is no test that a stale-version DB is detected/rebuilt — and there can't be, because the code doesn't do it. Test the append-only guard directly (re-insert same variation should conflict, `transcribe.py:78-86`). The DuckDB single-writer lock behavior under concurrent CLI + viewer access is untested and undocumented.
- **For Security (05):** path-traversal guards exist (`models.is_suspicious_entry` for zip extraction `reader.py:55-59`; `safeJoin` in `download.ts:79`; `..` checks in `index.ts:52,142`) — verify they're complete (symlink escape, encoded traversal). The `uv run` shell-out interpolates `${CASE_DIR}`/`${pageId}` into `Bun.$` (tagged-template-escaped, likely safe, but worth confirming).
**Critical:** No build blocker. The one architectural integrity risk to weight in synthesis: the mirror-immutability invariant — the project's entire value proposition — is enforced *structurally only at the web/export boundary*; past that boundary it is convention, and the schema-versioning no-op means a silently stale DB can misreport its own provenance. For a "provenance you can trust" engine, "the record is immutable because the writers choose not to touch it, and the version field is never checked" is the gap between the promise and the enforcement.
