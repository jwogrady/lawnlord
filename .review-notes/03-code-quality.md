## Finding

This is unusually disciplined code for an alpha. The working tree is clean, so this is a surface inspection rather than a diff review (the native /code-review had nothing to act on). What it surfaces is a codebase that reads like it was written by one person with a strong, consistent style: every module opens with a docstring that states its *job* and its *invariant*; every public function has a docstring that explains the contract and the *why* (ADR cross-references, the rationale for a constant, the failure mode being guarded); helper functions are small and single-purpose; and naming is precise and domain-faithful (`_anchor_index`, `_consensus_index`, `divergence_spans`, `escalate_case`, `installed_vision_models`). Comments overwhelmingly explain WHY, not WHAT — the standout examples are the magic-cutoff justification at `transcribe.py:28-33` (it tells you the working case shows a clean char-count gap so the exact threshold is insensitive), the repeat-penalty rationale at `transcribe.py:323-329` (a real bug — the "§" divider loop — and the verified fix), and the WSL-aware tuning notes. There are **zero** TODO/FIXME/XXX/HACK markers in either `src/` or `web/`, and magic numbers are properly hoisted into named module constants with explanatory comments (`MIN_PDF_TEXT_CHARS`, `RENDER_DPI`, `AGREEMENT_PRECISION`, `*_FLAG_THRESHOLD`, the `DEFAULT_*_HOST` ports). The web side is equally careful: `app.ts` is 828 lines of plain TypeScript with no framework, and it type-checks **clean under strict mode** (`bunx tsc --noEmit` → exit 0, zero errors).

The objective tooling story is mixed but reassuring once you read past the headline number. `ruff check` reports **65 violations, but 64 of them are F401 "imported but unused" and 63 of those are in a single file — `__init__.py` — which is the deliberate flat re-export surface the architecture review praised.** That is a false positive: the fix is one line in `pyproject.toml` (`[tool.ruff] ... per-file-ignores = {"__init__.py" = ["F401"]}`, or add `__all__`), not 63 edits. The two genuine findings are a real unused import and a semicolon-joined statement, both in `tests/test_transcribe.py:498,526` — trivial. `ruff format --check` says 16 of 26 files "would reformat," but the diff is **purely cosmetic line-wrapping**: the author writes at a ~99-column width with hand-aligned dict literals (`transcribe.py:167-169`) and joined SQL string-concatenation, and ruff's default 88-column would rewrap them. Setting `line-length = 99` in config would erase most of it. So the "no linter/formatter installed" gap the project map flagged is real *process* debt — nothing runs automatically, there is no CI, and a regression would not be caught — but the *current* code is already close to clean.

The genuine quality debt is complexity and a couple of fragile spots. Ruff's C901 flags 5 functions over the cyclomatic-10 line. The worst by a wide margin is `cli._main` at **complexity 34** (`cli.py:332-583`) — a 250-line `if args.command == ...` dispatch chain of ten branches, each opening a DB, running a command, building a Rich table, and printing. It is *readable* (flat, repetitive, each branch self-contained) but it is the obvious quick-win refactor: a dispatch dict mapping command → handler function would drop it to near-zero and make each subcommand independently testable. `transcribe.transcribe_case` (`transcribe.py:501-624`) isn't C901-flagged but is the real cognitive hotspot — a 120-line two-phase function (sequential pre-pass building a work queue, then a concurrent `ThreadPoolExecutor` phase committing per-future) with nested loops, per-variation rev logic, and three skip/fail accumulator lists; it's correct and well-commented but dense, and the per-variation `_next_rev` SQL call inside the page×transcriber double loop (`transcribe.py:582`) is an N×M query pattern. `export._rollup` (C901 12, `export.py:490-580`) packs coverage, agreement, fidelity-distribution, and flag-reason logic into one pass. The single most fragile piece of code in the repo is `_parse_local_output` (`transcribe.py:200-226`): regex salvage over malformed/truncated JSON with a chained `.rstrip().rstrip("}").rstrip().rstrip('"')` fallback — it works, it's commented, but it's brittle string surgery that deserves the heaviest test coverage and is exactly the kind of thing that breaks silently on a new model's output.

## Scoring

**Dimension:** Code readability / maintainability
**Score:** 8
**Rationale:** The code itself is genuinely high quality — disciplined single-author style, WHY-not-WHAT comments, hoisted constants, zero marker debt, precise domain naming, and a web layer that type-checks clean under strict mode. It loses points for *process*, not *code*: nothing is automated (no ruff/mypy config, no CI), the one true complexity offender (`_main` at C901 34) and the dense `transcribe_case`/`_parse_local_output` pair are real maintainability liabilities, and the unconfigured tooling means trivially-fixable lint/format noise sits uncaught. Fix the config (per-file-ignore + line-length) and the headline "65 violations" collapses to 2 real ones — so this is an 8 held back by missing guardrails, not bad code.
