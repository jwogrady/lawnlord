## Finding

The suite is far healthier than the README's "27 tests" claim suggests: `uv run pytest -q` collects and passes **94 tests in 4.82s**, and measured line coverage (`uv run --with pytest-cov pytest --cov=lawnlord`) is **83% (1451 stmts, 243 missed)**. The tests are well-balanced for a single-author alpha — characterization-style but genuinely exercising the hard seams. The fragile code Agent 03 flagged is, contrary to expectation, *well* covered: the local JSON-salvage path, the truncation→fidelity-0.0 safeguard (`test_transcribe.py:355-372`), exponential-backoff retry with transient-vs-permanent classification (`:236-262, :493-503`), concurrent-page ordering (`:217-225`), the partial-failure `failed` accumulator (`:293`), the per-variation append-only `force` rev guard (`:103`), Ollama-server-down degradation (`:397`), and the page-count-mismatch surfacing (`test_explode.py:92-104`) all have direct assertions. The vision backends are fully mocked — `_FakeClient` and a monkeypatched `urllib`/HTTP stub mean **no test makes a real network call**, which is correct for determinism. The big structural gap is acceptance testing: despite a real 17 MB intake zip and extracted case sitting in `intake/odyssey-250914566/` (manifest + 60+ real PDFs), **zero tests reference it** — every test builds a synthetic 1–2-page fixture. There is no end-to-end test that the actual court export ingests, explodes, and re-imports byte-identically, which is precisely the invariant the whole project sells.

Two real correctness/reliability gaps surfaced. **First, the cloud transcriber has no truncation safeguard.** `transcribe_page` (`transcribe.py:100-131`) calls `max_tokens=8000`, then does `json.loads(text)` on the response with no guard — if Claude hits the token ceiling on a dense page, the JSON never closes and the call raises an *unhandled* `JSONDecodeError` that propagates up as a hard failure (no `finish_reason`/`stop_reason` check, no salvage). The local (`_parse_local_output`, `:200-226`) and llama.cpp (`:344-350`, `finish_reason=='length' → fidelity=0.0`) paths both handle this gracefully; the cloud path — the *default, highest-trust* tier — does not, and no test exercises a truncated cloud response. **Second, observability is essentially absent for production debugging.** There is no `logging` import anywhere in `src/` — all output goes through one Rich `console.print` singleton aimed at a human terminal, with no levels, no timestamps, no structured context, and nothing written to a file. When a 4-hour overnight transcription run drops 12 pages into `failed`, the operator gets a count and a list of page IDs but no per-page reason, no model, no exception text, no retry trail. On the security front this is at least clean: the Anthropic key is only ever read from the env (`transcribe.py:90`, `cli.py:261`) and never printed, so there is no secrets-in-logs risk to hand off.

On resilience the core engine is solid. Server-down is handled by design — `installed_vision_models` returns `[]` and `ollama_available` returns `False` rather than raising (`transcribe.py:386-399`, tested at `:397`), so a missing Ollama/llama.cpp server degrades to "no local backend" instead of crashing. Concurrency is safe: the `ThreadPoolExecutor` runs only the network-bound `.transcribe` calls in parallel, and **every DuckDB write stays on the calling thread** (`transcribe.py:597-612`), committed per-future as it completes, so there is no multi-writer race inside a run and progress is durable mid-run (DuckDB autocommits each `execute` — there is no explicit `BEGIN`/`commit()` anywhere in `src/`, grep-confirmed, so the "durable mid-run" claim rests entirely on per-statement autocommit). The genuine reliability risks are at the boundaries: (1) DuckDB is single-writer-per-file, so a `transcribe`/`explode` run while the web viewer or a second command holds a handle will lock — untested, undocumented (Agent 02 flagged this; still true). (2) The web `/api/case` and `/api/exploded` routes have **no try/catch** (`web/index.ts:63-82`) while `/api/metrics` and `/api/regions` do — a `uv run` subprocess nonzero exit (locked DB, missing case, broken env) on those two routes becomes an unhandled `Bun.$` rejection and an opaque 500, taking down the *primary* lenses. (3) Every web request cold-starts a `uv run` Python process; if `uv` isn't on PATH or the env is broken, the entire viewer fails per-request with no health check.

- **Cloud truncation gap (untested, unhandled):** `transcribe_page` (`transcribe.py:100-131`) — `max_tokens=8000` then bare `json.loads(text)` at `:126` with no `stop_reason` check; a truncated cloud response raises an unhandled `JSONDecodeError`. - **`_parse_local_output` IS covered (better than 03 feared):** the `rstrip`-chain salvage (`transcribe.py:221`) and the no-`transcription`-key fallback (`:218`) — coverage shows only `:213-226` partially missed; the truncation/fidelity-0 behavior is directly asserted (`test_transcribe.py:371`).
- **Retry/backoff covered:** `_transcribe_with_retry` (`transcribe.py:447-463`, exp backoff + jitter) tested transient-retry-then-succeed (`test_transcribe.py:236-262`) and non-transient-no-retry (`:280`); `_is_transient` classification table tested (`:493-503`, covers 529/503/404).
- **Concurrency safety:** vision calls run in `ThreadPoolExecutor` (`transcribe.py:597`); every `con.execute` write is on the calling thread inside `as_completed` (`:603-612`) — no concurrent DB writes. Page-order independence asserted (`test_transcribe.py:217-225`). `escalate_case` (`:627`) uses the same pattern.
- **Durability via autocommit only:** `grep -rn "BEGIN\|\.commit()\|TRANSACTION\|rollback" src/lawnlord/` → **0 hits**. The "each unit committed the moment it finishes" claim (`transcribe.py:527,593`) relies on DuckDB per-statement autocommit, not an explicit transaction. No crash-recovery test (kill mid-run, re-run, assert resume).
- **No `logging`:** `grep -rn "import logging\|logging\.\|logger" src/lawnlord/` → **0 hits**. Sole output is the Rich `console` singleton (`console.py:9`). `failed`/`skipped` are returned as page-ID lists with no per-item reason (`transcribe.py:616-624`).
- **No secrets in logs (clean):** `ANTHROPIC_API_KEY` only read via `os.environ`/SDK default (`transcribe.py:90`, `cli.py:261`); never `print`/`console`-ed. Nothing to flag.
- **Server-down resilience (good, tested):** `installed_vision_models` returns `[]` on unreachable server, never raises (`transcribe.py:386-399`); `ollama_available` False (tested `test_transcribe.py:397`). CLI prints a `[yellow]` opt-in hint when no key + no models (`cli.py:282`).
- **Web error-handling asymmetry:** `/api/metrics` and `/api/regions` wrap the subprocess in try/catch → `{error}`+500 (`web/index.ts:91-103,113-124`); `/api/case` and `/api/exploded` do **not** (`:63-82`) — subprocess failure = unhandled rejection / opaque 500 on the primary lenses.
- **Type safety:** no mypy configured (Agent 00/03). Runtime holes are mostly guarded by tests, but the cloud `next(b.text for b in resp.content if ...)` generator (`transcribe.py:125`) raises `StopIteration` (uncaught) if a response has no text block — untested edge.

## Scoring

**Dimension:** Testing / production reliability
**Score:** 7
**Rationale:** 94 passing tests at 83% coverage with the fragile JSON-salvage/retry/concurrency seams genuinely well-asserted is strong for an alpha, and the core engine degrades gracefully when backends are down. It is held back from 8+ by three real gaps: no acceptance test against the real intake zip that sits *right there* (the project's central invariant goes unverified end-to-end), an unhandled truncation path on the *default* cloud tier that the local/llamacpp tiers both guard, and effectively zero production observability (no logging, failures reduced to bare ID lists). `cli.py` at 57% and `__main__.py` at 0% leave the user-facing dispatch lightly tested.
